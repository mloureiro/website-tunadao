# Deployment Guide

## Overview

This project consists of two parts:

- **Frontend (Astro)**: Deployed to GitHub Pages
- **CMS (PayloadCMS)**: Deployed to Render.com

### Data Architecture

The frontend is a **static site** (SSG) that fetches data from the CMS **at build time**:

1. During `npm run build`, Astro fetches all content from PayloadCMS REST API
2. Pages are pre-rendered with the fetched data
3. The generated static HTML is deployed to GitHub Pages
4. **No runtime CMS dependency** - the static site works without the CMS running

**Fallback mechanism**: If the CMS is unavailable during build, the frontend falls back to static data in `app/data/` to ensure builds always succeed.

## Frontend - GitHub Pages

### Automatic Deployment

The frontend automatically deploys to GitHub Pages when you push to the `main` branch.

### Manual Setup (one-time)

1. Go to your GitHub repository settings
2. Navigate to **Pages**
3. Under **Build and deployment**, select:
   - Source: **GitHub Actions**
4. Push to `main` branch to trigger the first deployment

### Custom Domain (optional)

1. Add a `CNAME` file to `/public/` with your domain
2. Configure DNS with your domain provider
3. Enable HTTPS in GitHub Pages settings

## CMS - Render.com

### Setup

1. Create a [Render.com](https://render.com) account
2. Connect your GitHub repository
3. Create a new **Web Service** from the `cms/` directory

### Configuration

**Build Command:**

```bash
npm install && npm run build
```

**Start Command:**

```bash
npm run payload -w cms -- migrate --force-accept-warning && npm run start
```

Database migrations are applied automatically on every deploy, **before the service begins serving
traffic**. The migrate step is idempotent — a redeploy with no new migrations is a no-op. If
migrate fails (schema mismatch or DB unreachable), `npm run start` is not reached and the deploy is
marked unhealthy, which is the correct fail-closed behaviour.

**Root Directory:**

```
cms
```

> only; it no longer gates the adapter. See the "Database schema & migrations" section below for the

> **Secret grep gate:** no real secret values are committed in source. Run
> `git grep -n "dev-secret-key-change-in-production" -- ':!wiki' ':!DEPLOY.md'` — must return nothing.

### Database - Turso

1. Create a [Turso](https://turso.tech) account
2. Create a new database:

   ```bash
   turso db create tunadao
   ```

3. Get the connection URL:

   ```bash
   turso db show tunadao --url
   ```

4. Create an auth token:

   ```bash
   turso db tokens create tunadao
   ```

5. Set `TURSO_DATABASE_URL` and `TURSO_AUTH_TOKEN` in the Render dashboard (or via `cms/render.yaml`).

### Database schema & migrations

The database schema is owned by **committed Payload migrations** in `cms/src/migrations`. Push is
**off in all environments** — `payload.config.ts` sets `push: false` unconditionally. Schema is
never auto-created from collection definitions at runtime.

**On every Render deploy** the start command runs:

```bash
npm run payload -w cms -- migrate --force-accept-warning && npm run start
```

This applies any pending migrations to Turso before the service serves traffic. If the schema is
already up to date the migrate step is a no-op. If it fails, the deploy is aborted (fail-closed).

> **R1 note:** migrations are verified against a local libSQL file DB (same `@libsql/client`
> that targets Turso). Remote Turso DDL apply is confirmed on first deploy.

**`PAYLOAD_DISABLE_PUSH`** is kept in `render.yaml` as a documentation marker only. The variable
no longer gates the adapter — push is disabled in code regardless of its value.

#### Developer schema-change workflow

When you add or change a collection or global:

1. Edit the collection/global definition in `cms/src/collections/` or `cms/src/globals/`.
2. Generate a migration:
   ```bash
   npm run migrate:create -w cms -- <descriptive_name>
   ```
3. Review the generated SQL in the new `cms/src/migrations/<timestamp>_<name>.ts` file.
4. Commit both the `.ts` and `.json` snapshot files alongside your collection change.

CI runs a drift check (`npm run migrate:check -w cms`) that fails the PR if a migration is missing
for a schema change. To check locally before pushing:

```bash
npm run migrate:check -w cms
```

To inspect the status of applied migrations on your local DB:

```bash
npm run migrate:status -w cms
```

#### Local setup (fresh DB)

If you have a fresh or empty local SQLite DB, apply migrations before seeding:

```bash
npm run payload -w cms -- migrate
npm run seed -w cms
```

Or use the destructive reset→migrate→seed round-trip (drops all local data):

```bash
npm run db:reset-and-seed -w cms
```

### First admin / first login

**When to run:** Once, after the Turso database, the Render environment variables, and the
migrations are all in place — and before anyone needs access to `/admin`. Run it from the repo root
with the required env vars exported in your shell.

**Why a script and not the `/admin` first-user screen:** The provisioning helper creates the first
admin user idempotently from a trusted shell instead of the public `/admin` first-user endpoint.
This avoids a brief window where the public screen is reachable before any admin account exists.

**Required env vars** (export in the shell before running):

| Variable              | Description                        |
| --------------------- | ---------------------------------- |
| `ADMIN_EMAIL`         | Email for the first admin account  |
| `ADMIN_PASSWORD`      | Password for the first admin       |
| `ADMIN_NAME`          | Display name (optional — default: `Admin`) |
| `TURSO_DATABASE_URL`  | Production Turso DB URL            |
| `TURSO_AUTH_TOKEN`    | Production Turso auth token        |
| `PAYLOAD_SECRET`      | **Must match the Render value** — see note below |
| `FRONTEND_URL`        | **Must match the Render value** — required by the production config (CORS/CSRF guard) |

> **`PAYLOAD_SECRET` note:** `PAYLOAD_SECRET` signs session **JWTs**; it does **not** hash
> passwords. Passwords are stored as `pbkdf2(password, per-user-salt)` and verify independently of
> the secret. The secret must still match the Render value so that sessions issued during
> provisioning and by the running web service are mutually valid. Provisioning now runs under the
> real `NODE_ENV=production`, so the config **requires** both `PAYLOAD_SECRET` and `FRONTEND_URL`
> — export the Render values before running.

**Command:**

```bash
export ADMIN_EMAIL=admin@example.com
export ADMIN_PASSWORD=$ADMIN_PASSWORD          # set to a strong bootstrap password
export ADMIN_NAME="Admin"
export TURSO_DATABASE_URL=libsql://your-db.turso.io
export TURSO_AUTH_TOKEN=$TURSO_AUTH_TOKEN      # from: turso db tokens create tunadao
export PAYLOAD_SECRET=$PAYLOAD_SECRET          # copy from Render dashboard → Environment tab
export FRONTEND_URL=https://mloureiro.github.io/website-tunadao  # copy from Render dashboard

npm run provision:admin -w cms
```

**What it does:**

1. Prints a masked target host (scheme + host only — token and credentials are never echoed).
2. Connects to the already-migrated DB.
3. Creates the admin user via the existing idempotent seeder.
4. Exits 0 on success.

**Data-safety and idempotency:** The command is safe to re-run. If the admin user already exists,
it logs `Skipped: … already exists` and exits 0 — it never duplicates or overwrites an existing
account. If `TURSO_DATABASE_URL` or `ADMIN_*` are missing (and `PROVISION_ALLOW_LOCAL` is not
set), the command fails immediately with a clear error and writes nothing to any database.

**Verification-only flag (`PROVISION_ALLOW_LOCAL=true`):** Setting this flag allows running the
script against a local throwaway SQLite file DB instead of Turso — useful for verifying
idempotency and fail-fast behavior without production credentials. The local DB must have
migrations applied first (`npm run payload -w cms -- migrate`). **Never use this flag for
production provisioning.**

```bash
# Apply migrations to local DB first
npm run payload -w cms -- migrate

# Local verification only — NOT production
ADMIN_EMAIL=test@example.com ADMIN_PASSWORD=$ADMIN_PASSWORD \
  PROVISION_ALLOW_LOCAL=true npm run provision:admin -w cms
```

**After first login:** Log in at `/admin` and change the password in the user profile UI if
`ADMIN_PASSWORD` was a temporary bootstrap value.

## Automatic Frontend Rebuilds

The CMS automatically triggers a frontend rebuild when content is published or updated.

### How It Works

1. When you save content in the CMS (blog posts, pages, videos, etc.)
2. PayloadCMS `afterChange` hooks detect the change
3. The CMS calls GitHub Actions API to trigger the `ci.yml` workflow
4. GitHub Actions rebuilds and deploys the static site to GitHub Pages
5. Changes appear live in ~2-5 minutes

**Coalescing**: Rebuilds are dispatched immediately. Rapid successive edits are coalesced by GitHub Actions concurrency (`cancel-in-progress` on `ci.yml`), so only the latest run proceeds.

### Setup

1. **Create a GitHub Personal Access Token:**
   - Go to GitHub → Settings → Developer settings → Personal access tokens → Fine-grained tokens
   - Create token with these permissions for your repo:
     - `Actions: Read and write`
   - Copy the token

2. **Add Environment Variables in Render:**

| Variable       | Description            | Example               |
| -------------- | ---------------------- | --------------------- |
| `GITHUB_TOKEN` | GitHub PAT from step 1 | `ghp_xxxx...`         |
| `GITHUB_OWNER` | GitHub username or org | `mloureiro`           |
| `GITHUB_REPO`  | Repository name        | `website-tunadao`     |

3. **Verify:**
   - Edit a blog post in CMS
   - Check CMS logs for `[Rebuild] Triggered...`
   - Check GitHub Actions tab for new workflow run

### Troubleshooting

- **"Skipping - GitHub credentials not configured"**: Environment variables not set
- **"Failed (403)"**: Token doesn't have Actions permission
- **"Failed (404)"**: Repository or workflow not found

### Alternative: Scheduled Rebuilds

If you prefer not to use webhooks, add a `schedule:` entry to the existing `on:` block in
`.github/workflows/ci.yml` (alongside the existing `pull_request`, `push`, and
`workflow_dispatch` triggers):

```yaml
# .github/workflows/ci.yml
on:
  pull_request:
    branches: [main]
  push:
    branches: [main]
  workflow_dispatch:
    inputs:
      deploy_target: # ...existing input definition...
  schedule:
    - cron: '0 */6 * * *' # Every 6 hours
```

## Environment Checklist

### Production

- [ ] GitHub Pages enabled in repository settings
- [ ] Render.com web service created
- [ ] Turso database created and connected (`TURSO_DATABASE_URL` + `TURSO_AUTH_TOKEN` set)
- [ ] Migrations applied on first deploy (automated via start command — verify in Render logs)
- [ ] `PAYLOAD_DISABLE_PUSH` kept in `render.yaml` as documentation marker (push is off in code)
- [ ] `FRONTEND_URL` set to production frontend URL (required for CORS/CSRF and provisioning)
- [ ] `PAYLOAD_SECRET` auto-generated by Render (verify in Environment tab)
- [ ] Cloudinary configured (`CLOUDINARY_*` and `NEXT_PUBLIC_CLOUDINARY_*`)
- [ ] Resend API key configured (optional — for contact form emails)
- [ ] GitHub token configured (`GITHUB_TOKEN`, `GITHUB_OWNER`, `GITHUB_REPO`)
- [ ] Custom domain configured (if applicable)

### Local Development

```bash
# Terminal 1 - CMS (apply migrations first if DB is new/empty)
npm run payload -w cms -- migrate
npm run dev -w cms

# Terminal 2 - Frontend
npm run dev -w app
```

**Note:** In dev mode, pages fetch from CMS on-demand (when you visit them), not at startup. You can start the frontend first and the CMS later.

Local dev uses SQLite (`cms/data/tunadao.db`) by default — no Turso credentials needed. Since push
is off in all environments, you must run `npm run payload -w cms -- migrate` once to create the
schema on a fresh local DB.

## Frontend Environment Variables

The frontend uses these environment variables for CMS integration:

| Variable              | Description                          | Default                 |
| --------------------- | ------------------------------------ | ----------------------- |
| `CMS_URL`             | PayloadCMS API URL                   | `http://localhost:3000` |
| `USE_CMS`             | Enable CMS fetching                  | `true`                  |
| `USE_TEST_FIXTURES`   | Use static fixtures (CI/E2E builds)  | `false`                 |
| `FORCE_PROD_CMS`      | Require live CMS (throw if down)     | `false`                 |

See `app/.env.example` for the full list.

**For production builds:**

```bash
# Build with CMS data
CMS_URL=https://your-cms.onrender.com npm run build

# Build with fallback data only (no CMS)
USE_CMS=false npm run build
```
